## Bug Found and Fixed

### `extract_top_import_roots` missed crates used only via derive macros

**Root cause:** `extract_top_import_roots` in `src/protocol/conventions.rs` only counted
`ReferenceKind::Import` entries from tree-sitter parsing. Crates like `thiserror` that are
used exclusively via derive macros (`#[derive(thiserror::Error)]`) produce no import references
in the tree-sitter index, so they were invisible to the enrichment system.

**Fix:** Augmented `extract_top_import_roots` to also extract dependency names from Cargo.toml
and package.json manifest symbols (e.g., `dependencies.thiserror` → `thiserror`). Manifest
deps get a base count of 1 (so they rank below heavily-used imports but still appear in the
top-N list). Cargo crate names are normalized (hyphens → underscores).

**Files changed:**
- `src/protocol/conventions.rs` — `extract_top_import_roots` now unions import refs + manifest deps
- `src/protocol/tools.rs` — added `test_explore_concept_enrichment_from_cargo_manifest`

**Verification:** `cargo check` clean, `cargo test --lib` 1359 passed / 0 failed, `cargo build --release` clean.

### Test 5 re-evaluation: correct behavior, not a bug

The original expectation that `explore("serialization")` should show `Enriched with project imports: serde`
was incorrect. `serde` is already in the Serialization concept's `symbol_queries`, so the enrichment
filter correctly excludes it as a duplicate. No other enrichment candidates (`bincode`, `rmp`,
`ciborium`, `ron`) are present in this project's dependencies. Empty enrichment is the correct answer.
