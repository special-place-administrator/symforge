## Successes

### 1. project-context.md Eliminated an Entire Class of Failure
87 rules loaded before every story. Zero architectural violations across 11 stories. No `anyhow` in library code, no SpacetimeDB write attempts, no async ControlPlane methods, consistent `mod.rs` style, import grouping, derive conventions, and error boundary rules followed silently. This is the direct fix from Epic 1's "agents improvise when conventions aren't loaded" problem.

### 2. Build-Order Discipline Was Transformative
Every story had an explicit mandatory build order — domain types first, pure functions second, wiring last. Story 2.2 made `process_file` a pure function before any pipeline code existed. That pattern carried through the entire epic, keeping parsing logic out of concurrent async code.

### 3. Type-System-Enforced Design Composed Under Pressure
`is_terminal()` as universal gate, `LanguageId` exhaustive matching, `RunManager` as single-owner coordination point, atomic registry writes — these four decisions made Stories 2.7-2.10 compose correctly despite combinatorial complexity (cancellation × checkpoints × re-indexing × invalidation).

### 4. Review Process Was Genuinely Adversarial
Not rubber-stamping. ~10 high-severity bugs caught across the epic including idempotency ordering (2.9), missing pipeline launch (2.9), stale idempotency records (2.10), is_systemic() misclassification (2.2), circuit breaker not breaking spawn loop (2.2). "Reviewer proposes, human decides" gave the right control level without bottlenecking.

### 5. Self-Annealing Story Dev Notes
Each review made the next story's dev notes better. Story 2.3 warned about `is_systemic()` misclassification because Story 2.2's review caught it. Story-embedded "patterns to prevent" tables acted as informal checklists derived from prior learnings.

## Challenges

### Root Cause: One Problem, Four Symptoms
The dev agent optimizes for "does the main path compile and pass." Once the happy path works, it declares victory. No systematic "am I done?" verification step exists. This manifests as:

**Symptom 1: No-op / Sentinel Tests** (killed 3 times across epic)
Tests that `assert!(true)` or use conditional logic that silently passes. Gives false confidence in test count.

**Symptom 2: `is_systemic()` Misclassification** (Story 2.2 H1)
Single-file I/O failure classified as systemic → aborted entire pipeline. The instinct to treat all I/O errors as fatal is baked into how agents think about error handling.

**Symptom 3: Data Computed but Dropped** (Stories 2.4, 2.9)
`not_yet_supported` counts computed but not surfaced. Prior run ID discovered but not connected. Agent builds logic correctly, then forgets to wire the output.

**Symptom 4: Missing Tests for New Code Paths** (Stories 2.1, 2.3, 2.8)
New error branches or state transitions added without corresponding tests. Agent implements happy path test, skips failure path.

### Why Better Dev Notes Alone Won't Fix This
We already embedded review patterns into dev notes. Story 2.6's dev notes warned about no-op tests — and Story 2.6 still generated one. The agent reads warnings, acknowledges them, and falls into the same pattern under cognitive load. The fix must be a mandatory process gate, not advisory guidance.

## Key Insights

1. **Convention-loading prevents improvisation.** project-context.md is the single highest-impact artifact in the development process. Zero violations across 11 algorithmic stories proves it.
2. **Recurring review findings indicate a generation problem.** The reviewer catching the same bug shapes 3 times means the dev process needs a pre-review quality gate, not just better warnings.
3. **The blind spot profile shifts between epics.** Epic 2 (write-heavy) had different risks than Epic 3 (read-heavy: trust boundary violations, invalidation gate bypass, empty/missing/stale disambiguation). The self-audit checklist and project-context.md must be recalibrated per epic.
4. **Foundation decisions pay compound dividends.** Stories 2.1-2.3 established RegistryPersistence, RunManager, is_terminal(), and atomic writes. Stories 2.7-2.10 composed correctly because the foundation was right.

## Epic 3 Blind Spot Analysis (Winston)

Epic 3's read-heavy patterns introduce different risks than Epic 2's write-heavy patterns:

1. **Trust boundary violations** — Retrieval must verify blob_id integrity before returning content. Agent optimizing for "does search return results?" will skip verification.
2. **Invalidation gate bypass** — Invalidated/unhealthy repos must refuse retrieval. Agent testing happy path will skip rejection tests.
3. **Empty vs missing vs stale disambiguation** — Three different responses collapsed into generic empty result violates Epic 3 ACs.
4. **Registry read performance** — Every search/retrieval is a registry read. Acceptable in Epic 2 (infrequent reads), potentially problematic in Epic 3 at scale.
